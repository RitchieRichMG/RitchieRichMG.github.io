System design overview

Architecture that stays safe and scalable

TaskForge is built in clear layers so it’s easier to secure, test, and scale later. The goal is simple: make helpful suggestions while keeping the user in control.

Core goal

Reduce stress by helping users organize tasks automatically. After connecting email and calendar accounts, TaskForge can suggest actions like creating calendar events from emails, turning messages into reminders, or organizing schedules. The user always stays in control, and risky actions need approval.

Layers (simple explanation)

Layer What it does Why it matters
Client (Web/Mobile) Login, connect accounts, review suggestions Clear UX + user approval
API Gateway Login checks, routing, rate limits Protects the system
Orchestrator + Policy Engine Runs workflows + checks permissions Stops unsafe actions
Queue + Workers Runs longer tasks (email/calendar work) Fast UI and scalable load
Data + Security PostgreSQL, Redis, Secrets Manager, Audit Log Reliable + secure + trackable

Main risks found

  • Security: too much access to external APIs
  • Coupling: orchestrator doing too much
  • Scalability: worker retries causing duplicate actions

Mitigations (what we changed)

  • Least‑permission OAuth scopes
  • Split roles: orchestrator vs policy vs workers
  • Unique task IDs + status tracking (idempotency)
  • Audit log for every action

Example: “No duplicate actions” rule

The problem
If a worker retries, it could create two calendar events or send the same email twice.
The fix
Give every task a unique ID, store status in the database, and check before running again.
